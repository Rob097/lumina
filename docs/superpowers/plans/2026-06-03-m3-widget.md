# M3 — The Widget (`apps/widget`) — Implementation Plan

> **For agentic workers:** TDD each task (tests fail first). Steps use checkbox (`- [ ]`) syntax.
> Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to run it task-by-task.

**Goal:** Opening `test-store.html` with one `<script>` line runs the entire flow end-to-end against the
public API and shows a draggable before/after result with a working CTA; the app bundle stays **< 45 KB
gzipped**.

**Architecture:** A 2-file loader (immutable `widget.js` loader + content-hashed `widget.[hash].js` app).
All flow logic lives in a **framework-agnostic core** (`src/core`) — config/locale, API client, a
`LuminaController` state machine, status polling, i18n, image pipeline — unit-tested under happy-dom with
injected deps. A **thin Preact view** (`src/ui`) rendered inside a **Shadow DOM** only renders controller
state and calls its methods. The public `window.Lumina` API (init/open/close/configure/on/off/preload/
version) is buffered by a command queue (Segment/GA pattern) and mirrors events as `lumina:<event>`
`CustomEvent`s. Wire: `GET /widget/config` → `sign-upload` → direct PUT to R2 → `generate` → poll
`/widget/status/:id`. The wire contract (Zod schemas/types) is reused from `@lumina/shared`; no duplicated
types (HARD RULE #6). Widget only ever sees a `site_key`, presigned URLs, and a `generationId` (HARD RULE #2).

**Tech Stack (new for M3):** `preact` · `vite` + `@preact/preset-vite` · `vitest` + `happy-dom` ·
`@playwright/test` · `terser` (min). Runtime deps: `preact`, `@lumina/shared` (types/Zod). No `@supabase/
supabase-js` in the core bundle (see D21).

## Decisions (append to docs/DECISIONS.md)
- **D21 — Status transport is polling-primary + pluggable.** The in-bundle transport polls
  `GET /widget/status/:id` with capped backoff until a terminal state; a `StatusTransport` interface lets a
  Supabase Realtime transport be lazy-loaded as a *separate* chunk later. Rationale: `@supabase/supabase-js`
  (~35 KB gz) alone would blow the **< 45 KB** budget (HARD RULE #7), which wins where it conflicts with the
  Realtime preference; `/widget/status/:id` is the spec's designated fallback and is sufficient for v1.
- **D22 — Two-stage build: immutable loader + content-hashed app.** `build.mjs` builds the app first
  (`widget.[hash].js`), reads the emitted filename, then builds the loader with that URL injected via Vite
  `define` (`__APP_BUNDLE_URL__`), emitting the unhashed, year-cacheable `widget.js`. No runtime manifest
  round-trip.
- **D23 — Framework-agnostic core, thin Preact view.** All flow logic is pure/injectable in `src/core`
  (testable under happy-dom without rendering); `src/ui` Preact components only render controller state and
  call its methods. Keeps logic testable and the bundle lean.
- **D24 — Client image pipeline = downscale ≤ 2048 long-edge + EXIF-orientation fix + re-encode
  (WebP→JPEG).** Re-encoding through a canvas strips EXIF/GPS client-side (defense-in-depth for HARD RULE
  #9; the server strips again). `computeTargetSize`, `parseExifOrientation`, `pickEncoding` are pure & unit
  tested; the canvas `processImage` wrapper is covered by E2E.
- **D25 — Anonymous visitor id** is a client-generated UUID persisted in `localStorage`
  (`lumina_anon_id`), sent as `anonId` for per-visitor abuse caps (§3.9). Storage is injected for tests.
- **D26 — Bundle-budget gate in CI.** `scripts/check-bundle-size.ts` gzips the emitted app bundle and exits
  non-zero if > 45 KB; wired into `build` and the CI workflow so a regression can't merge.
